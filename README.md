# Hotel Manager

A simple, mobile-first hotel management platform (lightweight PMS + direct-booking engine)
for small, family-owned hotels, lodges, homestays and resorts in West Bengal / Eastern India.

Single Next.js app (App Router + TypeScript), PostgreSQL via Prisma, NextAuth (credentials +
roles), Tailwind, installable PWA, full Bengali + English i18n via next-intl.

## Stack notes

- **Next.js 16** — this project was scaffolded on a very new Next.js version with real
  breaking changes vs older docs/training data (`middleware.ts` → `proxy.ts`, fully-async
  `params`/`searchParams`/`cookies()`, no default fetch caching). See `AGENTS.md` before
  changing routing/caching/auth code.
- **Prisma 7** — requires an explicit driver adapter (`@prisma/adapter-pg`); there is no
  built-in query engine connection anymore. Config lives in `prisma.config.ts`, not
  `package.json`. The generated client lives at `src/generated/prisma` (gitignored,
  regenerated by `prisma generate` / `prisma migrate dev`).
- **Multi-tenancy**: every hotel is a tenant (`Hotel.id`). All data is scoped by `hotelId`
  and every request re-verifies auth + role via the DAL in `src/lib/auth/session.ts` —
  `requireSession()` / `requireRole()` for Server Components, `requireApiSession()` /
  `requireApiRole()` for Server Actions and Route Handlers. `src/proxy.ts` only does an
  optimistic cookie check (redirect signed-out users to `/login`); it is not the security
  boundary.
- **i18n**: locale is a cookie (`locale=bn|en`), not a URL prefix — there's no `/en/...`
  routing, which keeps it simple and avoids fighting with `proxy.ts` for a single middleware
  slot. Bengali is the default. Add new strings to both `messages/en.json` and
  `messages/bn.json`.
- **PWA**: hand-rolled `public/manifest.webmanifest` + `public/sw.js` (no `next-pwa`/`serwist`
  dependency). The service worker deliberately never caches API routes, Server Action POSTs,
  or dynamic pages — only hashed static assets and an offline fallback page — because this
  app deals in live room availability and payments.

## Setup

1. **Install Node 20+** and **PostgreSQL** (locally or via Docker).
2. Install dependencies:
   ```bash
   npm install
   ```
3. Copy the env file and fill in values:
   ```bash
   cp .env.example .env
   ```
   - `DATABASE_URL` — your Postgres connection string.
   - `NEXTAUTH_SECRET` — `openssl rand -base64 32`.
   - `APP_ENCRYPTION_KEY` — `openssl rand -hex 32` (encrypts each hotel's Razorpay key
     secret at rest; Razorpay itself is configured per-hotel inside the app's Settings,
     never at the platform level — the platform is never the merchant of record).
4. Run the initial migration (creates all tables):
   ```bash
   npm run db:migrate
   ```
5. Seed demo data (a demo hotel, room types, rooms, guests, a spread of bookings across
   every status/source, a GST invoice, a housekeeping task, and one login per role):
   ```bash
   npm run db:seed
   ```
6. Start the dev server:
   ```bash
   npm run dev
   ```
   Open http://localhost:3000 — it redirects to `/login`.

## Project status

Scaffolded so far: Prisma schema (all V1 entities), Postgres + migrations, NextAuth
credentials login with role/hotel-scoped sessions, Bengali/English i18n with a language
toggle, PWA shell, and demo seed data. The actual PMS modules (rooms, rate calendar, front
desk, bookings, guests, payments, invoices, housekeeping, reports, public booking page, iCal
sync) are being built next, one working vertical slice at a time.
